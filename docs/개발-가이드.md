# Daymon 개발 가이드

체계적으로 개발하기 위한 규칙·흐름 정리입니다.

---

## 1. 브랜치 전략

| 브랜치 | 용도 |
|--------|------|
| **main** | 배포용. 항상 빌드 가능한 상태 유지. 푸시 시 GitHub Pages 자동 배포. |
| **feature/기능명** | 새 기능 개발 (예: `feature/chat-ui`) |
| **fix/설명** | 버그 수정 (예: `fix/egg-position`) |

- **main에 직접 푸시**: 소규모 수정·혼자 개발 시 허용.
- **PR 사용**: 기능이 크거나 리뷰가 필요할 때 `feature/*` → main 머지.

---

## 2. 커밋 메시지 (Conventional Commits)

**형식**: `타입(범위): 한 줄 설명`

| 타입 | 의미 |
|------|------|
| **feat** | 새 기능 |
| **fix** | 버그 수정 |
| **docs** | 문서만 변경 |
| **style** | 코드 의미 변경 없음 (포맷, 세미콜론 등) |
| **refactor** | 리팩터링 |
| **chore** | 빌드·설정·기타 |

**예시**
```
feat(egg): 알 크기 1.5배, egg2 위치 조정
fix(layout): 모바일에서 알 잘림 수정
docs: PDCA 가이드 추가
chore: 대화 메뉴 일단 제거
```

- 본문(선택): 왜 바꿨는지 한두 줄.
- `docs/`·`README`만 바꾼 경우 `docs:` 사용.

## 3. 폴더·파일 구조

```
daymon/
├── src/
│   ├── components/   # UI 컴포넌트 (컴포넌트명.jsx + 컴포넌트명.css)
│   ├── assets/       # 이미지·정적 파일
│   ├── utils/        # 공용 유틸 (userStorage 등)
│   ├── App.jsx, App.css
│   ├── main.jsx
│   └── index.css     # 전역 스타일
├── server/           # API 서버 (Express)
├── docs/             # 프로젝트 문서 (가이드, PDCA, 배포 등)
├── .github/workflows # CI/CD (배포)
└── public/           # 정적 파일 (있을 경우)
```

- **컴포넌트**: 한 파일에 한 컴포넌트. 스타일은 같은 이름의 `.css`.
- **이름**: 컴포넌트·폴더는 PascalCase/camelCase 일관 사용.

## 4. 작업 흐름 (한 사이클)

1. **할 일 정하기** — PDCA Plan 또는 이슈/메모.
2. **작업** — 로컬에서 `npm run dev` (또는 `dev:all`)로 확인.
3. **커밋** — `타입(범위): 설명` 형식으로 메시지 작성.
4. **푸시** — `main` 푸시 시 자동 배포.
5. **확인** — 배포 URL에서 동작·레이아웃 확인.
6. **정리** — PDCA Check/Act 또는 문서 업데이트.

---

## 5. 품질·일관성

- **EditorConfig**: `.editorconfig`로 들여쓰기·줄끝 통일 (에디터가 자동 적용).
- **문서**: 변경 사항이 크면 `docs/` 또는 `CHANGELOG.md`에 요약.
- **PDCA**: 기능 단위·스프린트 단위로 `docs/PDCA-사이클-템플릿.md` 복사해 사용.

---

## 6. 배포

- **자동**: `main`에 푸시 → GitHub Actions가 빌드 후 GitHub Pages 배포.
- **수동 확인**: 저장소 **Actions** 탭에서 워크플로 성공 여부 확인.
- **URL**: https://daymon-source.github.io/daymon/

자세한 절차는 `docs/배포하기.md`, `docs/자동배포-체크리스트.md` 참고.
