# Daymon 프로젝트에 PDCA 적용하기

## PDCA란?

**Plan(계획) → Do(실행) → Check(확인) → Act(조치)** 반복으로 개선하는 방법론입니다.

| 단계 | 내용 |
|------|------|
| **Plan** | 목표·지표 정하고, 할 일·우선순위 정하기 |
| **Do** | 정한 대로 실행 (작은 단위로) |
| **Check** | 결과 측정·비교, 원인 분석 |
| **Act** | 표준화 또는 다음 개선 과제 정하기 |

---

## Daymon에 적용하는 방법

### 1. 주기 정하기

- **스프린트**: 1~2주 단위로 한 사이클
- **기능 단위**: 알 탭 완성, 채팅 개선 등 하나씩
- **배포 단위**: 배포 전후로 한 번씩

### 2. Plan (계획)

- **목표**: 이번 주기에 “무엇을 달성할지” 한 문장으로
  - 예: “알 화면 레이아웃이 모바일/데스크톱에서 안정적으로 보이게”
- **지표**: 성공 여부를 볼 수 있는 것
  - 예: “모바일에서 알이 잘리지 않음”, “배포 후 1주일 내 버그 0건”
- **할 일 목록**: 이번 주기에 할 작업 3~5개
- **완료 기준**: 각 작업을 “언제 완료로 볼지”

### 3. Do (실행)

- Plan에서 정한 작업만 진행
- 변경 사항은 커밋 메시지에 “무엇을 왜 했는지” 간단히 기록
- 예상과 다른 일이 생기면 메모해 두기 (Check에서 사용)

### 4. Check (확인)

- **목표 달성 여부**: 지표대로 됐는지 확인
- **실제 vs 계획**: “예상과 다른 점” 정리
- **원인**: 잘된 것/안 된 것 각각 “왜 그랬는지” 짧게 정리
- **도구**: 로컬 테스트, 실제 기기 테스트, 배포 후 확인

### 5. Act (조치)

- **잘된 것**: 다음에도 쓰기 위해 문서·체크리스트로 남기기
- **안 된 것**: 다음 Plan에 “개선 과제”로 넣기
- **반복**: 다음 주기 Plan을 세우고 다시 Do → Check → Act

## 한 사이클 예시 (알 화면 개선)

| 단계 | 예시 |
|------|------|
| **Plan** | 목표: 알이 모바일에서 잘리지 않게. 할 일: Monster를 app-frame 직계 자식으로 이동, top/left 조정, 배포 |
| **Do** | 코드 수정, 로컬/폰 확인, main 푸시 |
| **Check** | 폰에서 알 전체 보이는지, 데스크톱 비율 깨지는지 확인 |
| **Act** | “알 위치는 app-frame 직계, top 320px”를 문서에 적어 두고, 다음에는 “채팅 UX”를 Plan에 넣기 |
